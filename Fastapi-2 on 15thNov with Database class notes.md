## 2. Basic Imports
```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()
```
- `FastAPI` — used to create the API app object.
- `BaseModel` (Pydantic) — used for **schema validation** of incoming request data.

---

## 3. Pydantic Model + POST Endpoint
```python
class Student(BaseModel):
    id: int
    name: str
    age: int

@app.post("/student")
def create_student(stud: Student):
    return stud
```
- Pydantic validates incoming data **before** it reaches your function body.
  - Example: sending `id` as a float instead of `int` returns a **422 error** with details — the function never even executes.
- This validation-first behavior is why Pydantic is used everywhere (FastAPI, LangChain, LangGraph, CrewAI, etc.) — it has become a standard for input validation across modern Python libraries.

## 4. Testing an API — 3 Ways
| Method | Notes |
|---|---|
| **Postman** | External tool; you must manually configure method, URL, and JSON body. |
| **Swagger UI** (`/docs`) | Auto-generated by FastAPI; lists every endpoint automatically, no manual config needed. Click **"Try it out"** → fill fields → **Execute**. |
| **curl** | Command-line testing; Swagger UI provides a ready-made curl command for each endpoint (`-X POST`, header `Content-Type: application/json`, `-d` for payload/data). |

- A response of **status 200** confirms the API call worked.
- Swagger automatically discovers all endpoints because it's a built-in FastAPI feature; Postman/curl require you to supply the URL and payload manually since they're independent tools.

## 5. Saving Data to a File
```python
def save_student_to_file(data):
    with open("students.txt", "a") as f:
        f.write(f"{data.id}, {data.name}, {data.age}\n")

@app.post("/students")
def create_student(stud: Student):
    save_student_to_file(stud)
    return {"message": "Student data saved successfully"}
```
- `"a"` = append mode (keeps adding new lines); `"w"` = write mode (overwrites each time).
- **Gotcha covered in class:** converting the Pydantic object to a dictionary (`stud.dict()`) before passing it caused an error (`'dict' object has no attribute 'id'`). Fix: pass the Pydantic object directly and access `.id`, `.name`, `.age` on it — no need to convert to a dict first.

## 6. Connecting to a Real Database (NeonDB / PostgreSQL)

### Why a database instead of a text file?
- Real-world systems (login, sign-up, forget-password, e-commerce, etc.) all persist data through databases via APIs, not flat files.

### Setup Steps
1. Sign up at **Neon** (serverless Postgres, free tier, no card required).
2. Create a new project → get a **connection string** (contains username + password).
3. Use the **SQL Editor** in Neon's dashboard to create a table:
   ```sql
   CREATE TABLE student (
       id INTEGER PRIMARY KEY,
       name VARCHAR(100) NOT NULL,
       age INTEGER NOT NULL
   );
   ```
4. Install the Postgres driver:
   ```bash
   pip install psycopg2-binary
   ```

### Python Connection Code
```python
import psycopg2
from psycopg2.extras import RealDictCursor

db_url = "<your-neon-connection-string>"

def get_connection_url():
    conn = psycopg2.connect(db_url, cursor_factory=RealDictCursor)
    return conn  # must return the connection object!
```
- **Common bug hit in class:** forgetting the `return conn` line caused `NoneType has no attribute 'cursor'` errors downstream.
- `cursor` = a pointer used to execute SQL statements against the DB (available in almost every SQL driver).

### Insert Function + Endpoint
```python
def store_student_in_db(student):
    conn = get_connection_url()
    cursor = conn.cursor()
    insert_query = "INSERT INTO student (id, name, age) VALUES (%s, %s, %s)"
    cursor.execute(insert_query, (student.id, student.name, student.age))
    conn.commit()      # persists the data permanently
    cursor.close()      # close pointer
    conn.close()        # close connection
    return {"message": "Student data inserted successfully"}

@app.post("/students/db/insert")
def insert_student(student: Student):
    return store_student_in_db(student)
```
- Flow: **connect → get cursor → execute parameterized query → commit → close cursor → close connection**.
- Common errors debugged live:
  - Typo `psycopg2.co(n)nect` → `AttributeError`.
  - Wrong table name (`students` vs `student`) → `relation "students" does not exist`.
  - Missing `return` in `get_connection_url()` → `NoneType has no attribute 'cursor'`.

## 7. Exposing a Local API to the Internet (ngrok)
- Purpose: let others (classmates) hit your **local** FastAPI server running on `localhost:8000`.
- Steps:
  1. Download & unzip ngrok, sign up (Google login works), copy your **auth token**.
  2. Configure the token:
     ```bash
     ngrok config add-authtoken <your-token>
     ```
  3. Expose your local port:
     ```bash
     ngrok http 8000
     ```
  4. Ngrok gives a public `https://...` URL. Append `/docs` to reach Swagger UI from anywhere:
     ```
     https://<random-id>.ngrok-free.app/docs
     ```
- Note: free-tier ngrok URLs are **randomly generated each session** (not static) — a static/custom domain requires a paid plan.
- On Windows, antivirus/Defender may flag the ngrok installer — temporarily adjust security settings to allow the download if needed.

## 9. Full Source Code Reference

### 9.1 `main.py` (from the previous class — GET endpoints, path/query params, first POST examples)

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/")
def test():
    return {"message": "Hello, World!"}


@app.get("/sudh/test/dsfas/fsdfs")
def test1():
    return "my name is sudhanshu i use to teach data"

students  = {1:"akash",2:"rohit",3:"sachin"}

@app.get("/students")
def get_student():
    return students


@app.get("/students/{stud_id}")
def student_serach(stud_id:int):
    return {"id":stud_id, "name":students[stud_id]}



@app.get("/add_student")
def add_student(stud_id:int, name:str):
    students[stud_id] = name
    return students


@app.post("/add_student_diff")
def add_studetn_diff():
    students['new_id'] = "new_name"
    return students

from pydantic import BaseModel

class newdata(BaseModel):
    stud_id:int
    name:str

@app.post("/add_student_new_value")
def add_detudet_new_value(newdata:newdata):
    students[newdata.stud_id] = newdata.name
    return students
```

**Explanation, block by block:**

- `app = FastAPI()` — creates the application object; every route is registered on this `app` instance.
- `@app.get("/")` → `test()` — a root GET route that just returns a static JSON message. Good for a quick "is my server alive" check.
- `@app.get("/sudh/test/dsfas/fsdfs")` → `test1()` — shows that a route path can be any string/nesting you want; it doesn't need to be meaningful. Returns a plain string instead of JSON.
- `students = {1:"akash", 2:"rohit", 3:"sachin"}` — an in-memory Python dictionary acting as a **fake database**. This only lives as long as the server process is running; it resets on every restart (this is the exact limitation that motivates moving to a real database later in `test.py`).
- `@app.get("/students")` → `get_student()` — returns the entire dictionary as-is.
- `@app.get("/students/{stud_id}")` → `student_serach(stud_id:int)` — a **path parameter** (`{stud_id}` in the URL). FastAPI automatically converts the URL segment to an `int` and validates it. Looks up one student by ID and returns `{"id":..., "name":...}`.
  - Note: if `stud_id` isn't a key in `students`, this will raise a `KeyError` (500 error) — not handled in this version.
- `@app.get("/add_student")` → `add_student(stud_id:int, name:str)` — uses **query parameters** instead of path parameters (e.g., `/add_student?stud_id=4&name=Rahul`). Adds/overwrites an entry in the dict and returns the whole updated dict.
  - Using GET to *modify* data (adding a student) isn't RESTful best practice — GET is conventionally meant for read-only operations — but it's shown here for teaching purposes before POST is introduced properly.
- `@app.post("/add_student_diff")` → `add_studetn_diff()` — a POST route with **no input at all**; it hardcodes a new key `'new_id': 'new_name'` into the dict every time it's called. Demonstrates that POST doesn't strictly require a request body.
- `class newdata(BaseModel)` — a **Pydantic schema** with two fields: `stud_id: int` and `name: str`. This is the first real schema-validated POST body in the file.
- `@app.post("/add_student_new_value")` → `add_detudet_new_value(newdata:newdata)` — takes a validated `newdata` object as the request body, adds it to the `students` dict using `newdata.stud_id` as the key and `newdata.name` as the value, then returns the updated dictionary.
  - Note the reused name: the parameter `newdata` and the class `newdata` share the same identifier — it works in Python because parameter scope shadows the class name locally, but it's confusing style; using `class NewData(BaseModel)` (PascalCase) and `data: NewData` as the parameter would be clearer.

```python

from pydantic import BaseModel
import psycopg2
from psycopg2.extras import RealDictCursor

app = FastAPI()
#pip install psycopg2-binary
db_url = "postgresql://neondb_owner:npg_Q73BAyDgsCez@ep-divine-forest-a4w59ake-pooler.us-east-1.aws.neon.tech/neondb?sslmode=require&channel_binding=require"

class Students(BaseModel):
    id : int
    name : str
    age : int


def get_connection_url():
    conn = psycopg2.connect(db_url, cursor_factory=RealDictCursor)
    return conn

def save_student_to_file(data):
    with open("students.txt", "a") as f:
        f.write(f"{data.id}, {data.name}, {data.age}\n")


@app.post("/students")
def create_student(stud:Students):
    save_student_to_file(stud)
    return {"message": "Student data saved successfully"}


@app.post("/students/db/insert")
def store_student_in_db(student: Students):
    conn = get_connection_url()
    cursor = conn.cursor()
    insert_query = "INSERT INTO student (id, name, age) VALUES (%s, %s, %s)"
    cursor.execute(insert_query, (student.id, student.name, student.age))
    conn.commit()
    cursor.close()
    conn.close()
    return {"message": "Student data inserted into database successfully"}
```

**Explanation, block by block:**

- **Imports:**
  - `FastAPI`, `BaseModel` — same roles as before (app object, schema validation).
  - `psycopg2` — the PostgreSQL driver for Python; lets Python talk to a Postgres database (Neon is Postgres under the hood).
  - `RealDictCursor` (from `psycopg2.extras`) — a special cursor type that returns query results as Python dictionaries (`{"id":1, "name":"x", ...}`) instead of plain tuples, which is easier to work with and to serialize into JSON.
- `db_url = "postgresql://..."` — the full connection string copied from Neon's dashboard: contains protocol, username, password, host, database name, and SSL options. This is the "address + credentials" needed to reach the hosted database from your local machine.
- `class Students(BaseModel)` — the schema for a student record: `id` (int), `name` (str), `age` (int). This is reused across **both** endpoints below (file-save and DB-insert), showing that one Pydantic model can validate input for multiple different backends (file, database, etc.) — validation is independent of where the data ends up.
- `get_connection_url()` — opens and returns a live connection to Neon using the URL above. This function is called fresh at the start of every request that needs the database (a new connection per request, closed at the end of that same request).
- `save_student_to_file(data)` — opens `students.txt` in **append (`"a"`) mode** and writes one comma-separated line per call: `id, name, age`. Because it's append mode, previous entries aren't lost — the file just keeps growing with each new student.
- `@app.post("/students")` → `create_student(stud:Students)` — validates the incoming JSON against `Students`, then calls `save_student_to_file(stud)` to persist it to the text file, and returns a simple success message (not the data itself).
- `@app.post("/students/db/insert")` → `store_student_in_db(student: Students)` — the main database-writing endpoint:
  1. `conn = get_connection_url()` — opens a DB connection.
  2. `cursor = conn.cursor()` — gets a cursor (pointer) to execute SQL through that connection.
  3. `insert_query = "INSERT INTO student (id, name, age) VALUES (%s, %s, %s)"` — a **parameterized** SQL statement; the `%s` placeholders prevent SQL injection by letting psycopg2 safely substitute values rather than string-concatenating them.
  4. `cursor.execute(insert_query, (student.id, student.name, student.age))` — runs the insert with the three validated values from the request body.
  5. `conn.commit()` — makes the insert permanent in the database (without this, the change would be rolled back / not saved).
  6. `cursor.close()` then `conn.close()` — cleans up the pointer and the connection so resources aren't left open.
  7. Returns a success message.

## Key Takeaways
- **Pydantic** = validation layer only; it rejects bad input before your business logic runs. It doesn't matter whether you're saving to a file or a database — the validation step is identical.
- **Swagger UI** (`/docs`) is the fastest way to test a FastAPI app since it needs zero manual configuration, unlike Postman or curl.
- Database operations always follow the same pattern regardless of provider: **connect → cursor → execute → commit → close**.
- Almost all real-world digital systems (login, checkout, forms, etc.) are powered by APIs connected to databases behind the scenes.
